# ADR-0002: 単一解像度グリッドへの回帰とGPU-First実装方針

- Status: Accepted
- Date: 2026-03-01
- Deciders: Physics/Simulation
- Scope: 水ソルバ基盤、性能戦略、今後の粉体/剛体連成の実装順序
- Supersedes: ADR-0001 の「空間LoD + block multirate を先行導入する実装順序」

## Context

現行の block/空間LoD/時間LoD 経路は、正しさを維持しながら性能を出すための実装複雑度が高い。
特に以下がボトルネックになっている。

1. block境界処理（owner/support/ghost/移管）と再構築コスト。
2. CPU並列での同期・色フェーズ・直列テールに起因するwall短縮の限界。
3. 将来のGPU化を見据えると、CPU向け複雑化ロジックの多くが再設計対象になる。

一方、目標は「水、粉体、剛体連成を段階的に成立させる」ことであり、
水の本番経路を早期にGPUへ移す方が全体工数を抑えやすい。

## Decision Drivers

1. 実装複雑度を抑えつつ水の本番性能を改善すること。
2. GPU常駐データでCPU/GPU転送を最小化すること。
3. 将来の粉体/剛体連成へ拡張しやすいI/Fを先に固定すること。
4. 旧CPU実装の維持コストを最小化し、GPU経路へ一本化すること。

## Decision

以下を採用する。

1. 可変空間LoD block と block境界連成をいったん停止し、
   **単一解像度グリッド**へ回帰する。
2. 実行戦略は **GPU-First** とし、水ソルバからGPU本番経路を導入する。
3. 計算削減は空間LoDではなく **active tile** で行う。
4. 旧CPU版MLS-MPMは参照用途に限定し、不要になった時点で速やかに削除する。

## Detailed Policy

### 1) データ配置

- 粒子/格子データはGPU常駐。
- 毎ステップ全量readbackはしない。
- CPU readbackは統計量とデバッグ用途のみ。
- CPU側に同等の本番計算経路は持たない。

### 2) 1ステップ計算順序（GPU）

1. active tile 構築
2. active tile のみ grid clear
3. P2G
4. grid update（外力・境界）
5. G2P

### 3) 連成導入順

1. 水単体（water_drop成立）
2. 剛体連成（剛体境界データをGPU経路へ統合し、反力I/Fを固定）
3. 粉体連成

### 4) レンダリング

- まずデバッグ用オーバーレイで粒子を円表示し、物理挙動検証に使う。
- その後、ドット絵テクスチャ描画をGPUで実装する。
- テクスチャ描画はフラグメントシェーダーを第一候補とし、必要に応じてcompute前処理を併用する。

### 5) 既存CPUコード整理

- 既存CPU実装は期待挙動の参照にのみ使用してよい。
- GPU移行後に不要と判定したCPUコードは、先送りせずに削除してコードベースを整理する。

## Consequences

### Positive

1. 実装の主複雑性を block境界処理からGPUパイプライン実装へ集中できる。
2. 早期に本番経路をGPU化でき、CPU最適化の手戻りを抑えられる。
3. 旧CPU経路の早期削除で保守コストを抑えられる。
4. 将来の連成拡張でI/F設計を先行固定できる。

### Negative / Risks

1. 初期GPU実装でデバッグ難度が上がる。
2. active tile 構築・compaction実装の品質が性能を左右する。
3. 既存コード削除を急ぎすぎると、調査に必要な手掛かりを失う可能性がある。

## Non-Goals

- 可変空間LoD block の継続改善
- block彩色並列のさらなる最適化
- block境界フラックス交換の高精度化

## Validation Plan

最小受け入れは `water_drop`。

- GPU経路で安定実行できること（NaN/発散なし）。
- 質量誤差・侵入率が閾値内であること。
- pass別GPU時間を計測できること。
