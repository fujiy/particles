# Rendering Specification (codex)

本書は `design.md` のレンダリング章を具体化した実装仕様である。疑似コードは不要とし、データ表現・更新規約・しきい値・責務境界を明示する。

## 1. 目的と非目的

### 1.1 目的
- 2Dサンドボックスにおける **地形/流体/粒子**の可視化を、GPU常駐データを中心に実装する。
- 地形は **Near/Far の2層キャッシュ**により、極端なズーム/高速パンでも破綻（穴・フレーム落ち）を抑える。
- 「Nearではガチタイル（8x8px）」という美術要件を維持しつつ、サブピクセル領域ではモアレ/ちらつきを抑える。

### 1.2 非目的
- 本書はアート制作パイプライン（ドット絵の作り方）を規定しない。
- 物理ソルバの詳細（MLS-MPM の数式）は `physics.md` に委譲する。

## 2. 用語と座標系

- **セル (cell)**: 地形の論理最小単位。`CELL_SIZE_M`（m）で定義。
- **チャンク (chunk)**: セルの保存/編集単位。`CHUNK_SIZE` セルの正方領域。
- **タイル (tile)**: レンダ更新最適化単位。実装上はチャンクと一致してよいが、将来分離可能にする。
- **Near**: 近距離詳細レイヤ。材料IDで描画し、タイルアートを忠実に表示。
- **Far**: 遠距離背景レイヤ。材料集約/密度で表示し、Nearの穴埋めと超ズームアウトを担う。
- **LODレベル k**: サンプリングピッチが `2^k` 倍になる離散スケール。

座標は以下を明示的に分離する（`design.md` と整合）:
- `world_pos_m`: 連続空間（m）
- `global_cell`: セル格子上の整数座標
- `chunk_coord`: チャンク格子上の整数座標
- `local_cell`: チャンク内セル座標

## 3. レンダリング全体像（パス構成）

本ゲームは Bevy + wgpu を前提とする。

- **TerrainPass**
  - `TerrainNearUpdate` (compute): Nearキャッシュの dirty 矩形/タイルを更新
  - `TerrainFarUpdate`  (compute): Farキャッシュの dirty 矩形/タイルを更新
  - `TerrainCompose`    (fragment): Far→Near の順に合成して画面へ（必要ならデバッグオーバーレイもここ）
- **Particles/WaterPass**
  - 検証段階: CPU/Bevy標準で粒子円を描画（デバッグ）
  - 製品段階: GPUドット絵描画へ移行（別途仕様化）

以降は Terrain を主対象として規定する。

## 4. 地形表示の2モード

カメラズームにより、「セルが画面上で何pxか」を評価しモードを切り替える。

- `s = (CELL_SIZE_M * camera_zoom) / screen_pixel_size_m`
  - 直感: 「1セルが画面で何ピクセルに見えるか」

### 4.1 拡大モード（Magnify）
- 条件: `s >= 1` 付近
- 表現: **材料IDベース（ガチタイル）**
- フィルタ: 基本は最近傍（nearest）。境界線/ディザ等はアート側ルールに従う。

### 4.2 縮小モード（Minify）
- 条件: `s < 1` 付近
- 表現: **Farレイヤ優先**。Nearは品質上限/予算上限により早期に抑制。
- 目的: サブピクセルで発生するモアレ/ちらつきを抑える。

### 4.3 ヒステリシス
モード切替のチャタリングを防ぐため、2つの閾値を持つ。

- 縮小→拡大: `s > S_UP`（例: 1.2）
- 拡大→縮小: `s < S_DOWN`（例: 0.8）

`S_UP > S_DOWN` を必須とする。数値はチューニング可能だが、デフォルトをパラメータとして固定する。

## 5. キャッシュの器（解像度・アスペクト・範囲）

### 5.1 基本方針
- **器（テクスチャ/バッファの解像度）は固定**し、モードによって中身の生成・フィルタを変える。
- パンの追従はリングバッファで行い、「移動で増えた分だけ」更新する。

### 5.2 アスペクト比
- キャッシュの内部アスペクト比は固定（例: 16:9）。実ウィンドウが異なる場合は切り抜き（中央クロップ）またはレターボックスで対応する。
- 目的: リサイズ時にリング座標変換やリソース再生成が頻発しないようにする。

### 5.3 解像度上限（Near）
Nearは「縮小直前が最悪に重い」問題を避けるため、ユーザー設定で上限を持つ。

- `NearQuality = { Full(1/1), Half(1/2), Quarter(1/4) }`
- 実装上は Near のレンダターゲット（または地形用テクスチャ）を
  - `screen_res / q`（q∈{1,2,4}）に固定し、最終合成で拡大して表示する。

### 5.4 マージン（パン耐性）
- Near/Far ともに、画面に対して 1+α の範囲を保持する。
- Near: 例 `1.25〜1.5` 倍
- Far : 例 `2.0〜3.0` 倍（低解像なので広く取る）

マージンは「リング更新頻度」と「更新矩形サイズ」のトレードオフであり、パラメータ化する。

## 6. Near / Far のデータ表現

### 6.1 Near（材料ID）
Nearは「ガチタイル」を成立させるため、材料IDを主データとする。

- `near_material_id`: `u16` or `u32` の2D配列（セル/タイル単位）
- `near_variant`: オートタイル用のサブID（任意）
- 出力: `near_color_tex`（表示用テクスチャ、解像度は NearQuality に従う）

描画は2段階でも良い:
- (A) 直接 `near_color_tex` にタイルをラスタライズ（compute）
- (B) `near_material_id` を持ち、fragmentでタイルアトラス参照して描画（vertex+fragment）
(A) はドローコール削減に有利、(B) は実装が単純になりやすい。初期は (A) を推奨（CPU→GPU転送を完全に避けやすい）。

### 6.2 Far（集約/密度）
Farは「縮小時の安定表示」を目的に、集約した表現を持つ。

最低限の推奨表現:
- `far_top1_id: u16`
- `far_top2_id: u16`（任意）
- `far_w1: u8`（0..255、top1の被覆率）
- `far_w2: u8`（任意）
- もしくは `solid_fraction: u8`（地形/空の比率）

生成方法は以下のいずれか（選択）:
- **多点サンプルの多数決/ヒストグラム**（2x2〜4x4 など）
- **階層集約（mipmap的）**: LOD0(材料ID)→LOD1(集約)→...

Farの解像度は Near より低く（Nearの 1/2〜1/4）、広い範囲を持つ。

## 7. LOD（離散スケール）とズーム運用

### 7.1 LODレベル
- `k` は整数、サンプリングピッチは `pitch = 2^k`
- `Near` は現在のズームに対応する `k_near`
- `Far` は `k_far = k_near + FAR_OFFSET`（例: +3 → 8倍）を基本とする

### 7.2 ズーム時の更新方針（再サンプルの責務）

本仕様では **ズーム時に再サンプル（スケール切替）を行うのは Far のみ**とする。

- **Far**:
  - `k_far` が変化したタイミング（例: ズームが概ね2倍/半分を跨ぐ）で、Farキャッシュを「新しいピッチ」で再評価する。
  - スパイク抑制のため、次を併用してよい:
    - 旧Farからの即席リサンプル（縮小/拡大）で暫定表示を作る
    - dirtyキューに積んで予算内で逐次リファイン（全面を1フレームで作り直さない）
  - Farは背景・穴埋めが主目的なので、暫定表示の許容時間はNearより長くてよい。

- **Near**:
  - **器（解像度）は固定**であり、ズームに伴う「再サンプル（mipmap相当）」は行わない。
  - 代わりに、ズームにより **Nearがカバーすべきワールド領域（viewport in world）が変化**するため、
    - 新しい viewport に対応する **必要領域だけ**を dirty として更新する。
    - ズーム中心が画面中心に固定される操作系を想定すると、必要領域は主に **中央付近**に発生するため、更新矩形は通常小さい。
  - Nearの未更新領域は見えない（切り抜き）か、見えても Far が背後で穴埋めする前提でよい。

注: ズーム操作が「中心固定」ではなく、ピボットがずれる場合は、dirty矩形の中心をそのピボットへ移すだけでよい（考え方は同じ）。

## 8. パン（平行移動）とリングバッファ更新

### 8.1 リング座標
キャッシュは「論理座標系（world）」と「物理格納（texture）」を分離する。

- `cache_origin_world`: キャッシュ左上（または中心）のワールドセル座標
- `ring_offset`: 物理テクスチャ上のオフセット（mod幅/高さ）

カメラ移動により `cache_origin_world` がセル単位で変化した分だけ、
- 新規に可視になる行/列（またはタイル矩形）を dirty として発行する。

### 8.2 大移動（テレポート）時
`abs(dx)` や `abs(dy)` がキャッシュサイズの一定割合を超えた場合は、
- リング更新よりも **再初期化（全面dirty）** を選択してよい。
閾値はパラメータ化（例: 幅の 1/2）。

## 9. 更新スケジューリング（budget）

### 9.1 Dirtyキュー
Near/Far それぞれに dirty 更新キューを持つ。
- 単位はタイル（推奨）または矩形
- 優先度: 画面中心に近いほど高い / 直近で見えるほど高い

### 9.2 予算
1フレームあたりの上限を設け、スパイクを抑える。

- `max_tiles_per_frame_near`
- `max_tiles_per_frame_far`
- `max_pixels_per_frame`（矩形更新を使う場合）
- `max_compute_time_ms`（可能ならGPUタイムスタンプで制御）

Nearは品質要件が高いので、Farより優先。

## 10. 物理との整合（重要）

- **物理（接触/境界）は見た目LODに依存させない。**
- 地形生成関数（決定論ベース）+ 編集差分は、物理問い合わせでは常に「正」の解像度で評価する。
- レンダ側の Far 集約は、あくまで見た目の近似であり、当たり判定やSDF境界そのものにはしない。

## 11. Shader構成（Bevy / WGSL）

### 11.1 ファイル分割と共有コード
WGSL自体に `import/include` は無いが、Bevy のシェーダ資産では `#import` 相当の仕組みを使用できる前提で構成する。

## 12. デバッグと計測

- デバッグオーバーレイ:
  - Near/Far の更新矩形表示
  - LOD k の表示
  - キャッシュ原点・リングオフセット表示
- 計測:
  - Near/Far の更新タイル数
  - 更新に要したGPU時間（可能なら）
  - ドロップフレーム発生時の原因（予算超過、テレポート等）
